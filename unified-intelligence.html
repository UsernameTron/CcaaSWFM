Why MCP Abstraction
Direct API wrappers create N×M complexity. MCP lets consumers ask what they need — the server handles multi-platform extraction. Add a platform? One connector, zero consumer changes.
Agent ID Resolution
Genesys: UUID. UKG: employee number. Helpdesk: email. Redis cache maps any platform ID to canonical in <2ms. Without this, cross-platform joins are impossible.
Platform-Agnostic
Swap Genesys for NICE CXone, add connectors for Cognigy and Observe.AI — the MCP layer, ID resolution, and anomaly correlation remain identical.

from mcp.server import Server
from mcp.types import Tool, TextContent

server = Server("ispn-operations")

@server.tool()
async def get_daily_performance(date: str) -> list[TextContent]:
    """Fetch and correlate daily metrics across all platforms."""
    
    # Parallel extraction from all three sources
    calls, schedule, tickets = await asyncio.gather(
        genesys.get_conversation_metrics(date),
        ukg.get_schedule_adherence(date),
        helpdesk.get_ticket_summary(date)
    )
    
    # Normalize into unified performance model
    perf = DailyPerformance(
        date=date,
        calls=normalize_call_metrics(calls),
        staffing=normalize_schedule(schedule),
        tickets=normalize_tickets(tickets),
    )
    
    # Cross-platform correlation
    perf.anomalies = detect_cross_platform_anomalies(perf)
    perf.cost_per_contact = calculate_cpc(
        labor_hours=perf.staffing.actual_hours,
        call_volume=perf.calls.total_handled,
        blended_rate=19.50
    )
    
    return [TextContent(text=perf.to_report())]

    # From Helpdesk
    tickets_created: int
    escalation_count: int
    
    # Computed cross-platform
    utilization: float       # (handle_time * calls) / in_queue_sec
    cost_per_contact: float  # (hours * rate) / calls
    composite_score: float   # FCR(30%) + CSAT(25%) + AHT(20%) ...

def resolve_agent_id(platform: str, pid: str) -> str:
    """Genesys=UUID, UKG=emp_num, Helpdesk=email → canonical"""
    return redis_cache.get(f"{platform}:{pid}")  # <2ms p99

def detect_cross_platform_anomalies(perf: DailyPerformance) -> list[Anomaly]:
    anomalies = []
    
    aht_z = z_score(perf.calls.aht, perf.calls.aht_baseline)
    ticket_ratio = perf.tickets.total / perf.tickets.baseline_avg
    adherence = perf.staffing.adherence_pct
    volume_z = z_score(perf.calls.total, perf.calls.volume_baseline)
    
    # Pattern: AHT↑ + Tickets↑ + Adherence OK = SYSTEM ISSUE
    if aht_z > 2.0 and ticket_ratio > 1.5 and adherence > 85:
        anomalies.append(Anomaly(
            type="SYSTEMIC",
            detail="AHT + tickets elevated, adherence normal → system latency",
            signals={"aht_z": aht_z, "ticket_ratio": ticket_ratio}
        ))
    
    # Pattern: Volume↑↑ + SL↓ + Abandon↑ = CAPACITY
    if volume_z > 2.0 and perf.calls.service_level < 70 and perf.calls.abandon > 8:
        anomalies.append(Anomaly(
            type="CAPACITY",
            detail="Demand exceeds staffing model → Erlang C recalc triggered",
            action=trigger_erlang_recalc(perf)
        ))
    
    return anomalies

Why Multi-Signal
Single-metric alerts generate noise. AHT spikes happen legitimately. But AHT↑ + Tickets↑ + Adherence→ is a system latency signature. Cross-correlation turns alerts into diagnostic intelligence.
Erlang C Integration
When CAPACITY anomaly triggers, the engine automatically recalculates required staffing using live data — not stale forecasts. Real-time demand → real-time staffing model.
Action Routing
SYSTEMIC → platform team. STAFFING → WFM lead. CAPACITY → Erlang recalc + escalation. The classification determines who gets the alert, not just that there is one.
